1. 프로젝트 개요

1.1 배경 및 현황

(주)인시스 그룹은 아웃소싱 전문기업으로 직영 5개 사업장과 계열사 7개 법인, 경영지원 대상 업체 포함 총 29개사를 운영하고 있습니다. 현재 업무 전반이 엑셀 기반 수작업으로 운영되어 다음과 같은 문제가 지속되고 있습니다.

  • 데이터 분산: 계열사별 독립 관리로 인한 인사·급여·회계 데이터 불일치
  • 업무 비효율: 수작업 급여 계산, 수동 계산서 발행, 종이 서류로 인한 시간 낭비
  • 의사결정 지연: 경영진이 전사 현황을 실시간으로 조회할 수 없음
  • 보안 취약: 사내 중요 데이터가 외부 AI 서비스에 무분별하게 노출
  • 이력 관리 공백: 현장직·파견 근로자 과거 근무 이력이 엑셀로 분산 보관

1.2 플랫폼 전략

LG CNS 내부 ITSM 프로젝트 Vera(약 20만 라인)의 설계 지식을 기반으로, 클린룸 방식으로 SMB ERP 플랫폼을 새로 구축합니다. Vera 코드를 직접 참조하지 않고 아키텍처 설계 경험과 패턴 지식만 활용합니다. SMB 시장 범용 플랫폼으로 설계하되 insys 그룹을 첫 번째 고객으로 구현합니다.

1.3 핵심 원칙

원칙내용
메타데이터 기반테이블·필드·UI·로직을 메타데이터로 정의. 코드 변경 없이 업무 모듈 추가
AI Native자연어→SQL 분석, Vector 검색, Record Copilot 내장. 경영자 챗봇이 킬러 기능

5.2 동작 원리

  • 관리자가 No-Code Studio에서 테이블 생성
  • sys_table, sys_column에 메타데이터 저장
  • 실제 PostgreSQL 테이블 DDL 자동 실행
  • sys_form_layout, sys_list_layout 자동 생성 (AI Auto Layout)
  • sys_menu에 메뉴 항목 자동 등록
  • 사용자가 업무 포털에서 해당 메뉴 클릭
  • 프론트엔드가 메타데이터 조회 → 동적 화면 렌더링
  • 데이터 CRUD는 범용 API 엔드포인트로 처리

6.5 챗봇 빌더 + 경영자 AI 챗봇 (킬러 기능)

챗봇기능
경영자 챗봇'이번 달 전체 매출이 얼마야?' → AI가 SQL 생성 → 즉시 조회. 전 계열사 또는 개별 선택
경영지원실 챗봇급여·연차·휴가 자동 답변. 사내 DB에만 접근 (외부 유출 없음)
전략기획실 챗봇인력 요청·배치 현황 자동 답변
현장 챗봇작업지시·안전점검·출퇴근 확인 (모바일)

7. PostgreSQL DB 스키마 설계

7.1 스키마 구성 전략

플랫폼 코어(메타데이터)는 public 스키마에, 각 계열사 데이터는 개별 스키마에 저장합니다. 본사 통합 조회는 Cross-schema 읽기 전용 뷰로 구현합니다.
스키마대상주요 테이블
public플랫폼 코어sys_table, sys_column, sys_workflow, sys_role, sys_permission 등
common전사 공통users, audit_logs, notifications
insys인시스 본사employees, departments, work_history, payroll, vouchers...
pine(주)파인employees, payroll, staffing...
ire(주)이레employees, payroll, buildings...
chance(주)챈스trade_orders, customs, forex...
coresys(주)코어시스템employees, payroll, production...
기타 계열사4개 법인각 계열사별 독립 스키마
property건물 관리buildings, units, lease_contracts, sale_contracts, maintenance...
staffing인재·인력공급candidates, client_companies, staffing_contracts, dispatch...

7.2 보안 설계

  • 암호화: 이름·주민번호·계좌번호·전화번호·급여 금액 → AES-256 암호화
  • RLS: Row Level Security로 계열사 간 데이터 완전 격리
  • 감사 로그: 모든 조회·수정·다운로드 common.audit_logs 자동 기록
  • 외부 AI 차단: ChatGPT·Gemini 등 외부 도메인 네트워크 레벨 차단
  • 암호화 키: AWS KMS 관리 (코드에 하드코딩 금지)

10. 클린룸 규칙

원칙: 같은 설계자가 더 나은 버전을 처음부터 새로 만드는 것. 코드 유사성이 아닌 코드 독립성이 핵심.

10.1 허용

  • Vera에서 축적된 아키텍처 설계 지식·패턴·기술 판단 활용
  • '메타데이터로 동적 테이블을 만든다'는 개념적 접근
  • 'RBAC 6단계 스코프'라는 설계 판단
  • 'Workflow를 LangGraph로 구현한다'는 기술 선택

10.2 금지

  • Vera의 .py, .jsx 등 소스코드 파일 직접 참조
  • 함수·클래스·변수 구조를 복사 후 이름만 변경
  • Vera 코드를 열어놓고 구현하는 행위

10.3 클린룸이란?

원래 반도체 공장의 먼지 없는 무균 작업실에서 나온 개념입니다. 소프트웨어에서는 기존 코드의 오염 없이 완전히 새로 만드는 개발 방식을 의미합니다.

구분허용 예시금지 예시
개념·설계'메타데이터로 동적 테이블' 개념Vera의 metadata.py 직접 참조
코드동일한 목적의 코드를 처음부터 새로 작성Vera 함수를 복사 후 이름만 변경
법적 의미설계 경험은 개인의 지식재산코드는 LG CNS의 저작권 자산
